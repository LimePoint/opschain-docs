Introduction

OpsChain is an orchestration and change automation platform based in GitOps principles with a core objective to Unify Change. Connect, automate, and orchestrate people, processes, and tools across your on-premise and cloud platforms.

Projects

Projects are used to organise environments and changes and are analogous to JIRA projects. Each project has:

  • a Git repository containing configuration for that project
  • properties where you can store project specific:
    • key value pairs
    • environment variables and values (that will be available in the Unix environment running a change action)
    • files (that will be written to the working directory before running a change action)

The OpsChain CLI provides commands for interacting with projects via the opschain project command.

Environments

Environments represent the logical infrastructure environments under a project (for example development or production). Environments also have an associated set of encrypted properties. If a property exists at project and environment level, the environment value will override the project value.

The OpsChain CLI provides commands for interacting with environments via the opschain environment command.

To collaborate on an OpsChain project, you need to configure a Git remote for the project's Git repository. See Git Basics: Working with Remotes for more information regarding Git remotes.

  • Add a Git remote: opschain project add-git-remote
  • Archive a Git remote: opschain project archive-git-remote
  • Update the credentials of a Git remote: opschain project update-git-remote
  • List the Git remotes: opschain project list-git-remotes

Changes

A change is the application of an action from a specific commit in the project's Git repository to a particular environment. A step will be created for the change action, with additional steps created for each child action it requests.

Only one change can be running in an environment at a time. Changes will sit in the pending state whilst waiting for the existing change to finish.

The OpsChain CLI provides commands for interacting with changes via the opschain change command.

Log lines

By default, change log output is stored in the OpsChain database indefinitely. The log_lines endpoint allows you to retrieve the log output for further processing. See retention configuration for information on configuring OpsChain to remove historical change logs.

Change logs are also retrieved from oldest to newest. If you prefer to retrieve the newest log lines first, you can override the order using the sorts Ransack filter (e.g. filter[sorts]=logged_at desc). Note: this API only supports sorting based on the logged_at field in the ascending (logged_at asc) or descending (logged_at desc) direction - it can't be ordered by any other fields.

The OpsChain CLI allows you to view the change logs via the opschain change show-logs command.
